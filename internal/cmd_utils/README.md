# `cmd_utils` Test Scenarios

This document provides a summary of the test scenarios for the `cmd_utils` package, which is responsible for the low-level execution and piping of shell commands.

| Scenario                 | Description                                                                                                                                                                                    | Motivation                                                                                                                                                                                             | Utility                                                                                                                                                                  | Test Location                                                                                             | Test Type   | Dependencies | Execution Command                                                                                             |
| :----------------------- | :--------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :-------------------------------------------------------------------------------------------------------- | :---------- | :----------- | :------------------------------------------------------------------------------------------------------------ |
| **No Commands**          | Verifies that `PipeCmds` returns immediately with no output or error when given an empty list of commands.                                                                                      | This is a basic sanity check to ensure the function handles edge cases gracefully without panicking or causing unexpected behavior.                                                              | It guarantees that consumers of this function don't have to add their own checks for empty inputs, simplifying their code.                                                     | `TestPipeCmds/no_commands`                                                                                | Unit        | None         | `go test -run ^TestPipeCmds/no_commands$`                                                                     |
| **Single Command**       | Tests the execution of a single command, checking both successful output and error conditions.                                                                                                 | This ensures the function can act as a simple command runner, not just a piper, providing a consistent interface for all command execution.                                                      | It confirms that the function is a reliable replacement for a direct `exec.Command().Output()` call, but with the added benefit of being testable.                             | `TestPipeCmds/single_command_success`<br>`TestPipeCmds/single_command_error`                               | Unit        | None         | `go test -run ^TestPipeCmds/single_command`                                                                   |
| **Successful Pipe**      | Simulates a successful pipeline of two commands (e.g., `cmd1 \| cmd2`), ensuring the final output is correct.                                                                                    | This is the primary use case for the function. The test validates that the standard output of one command is correctly connected to the standard input of the next.                             | It proves that the core piping mechanism works, which is critical for chaining commands together to perform complex tasks, like filtering `git` output with `grep`.         | `TestPipeCmds/pipe_two_commands_success`                                                                  | Unit        | None         | `go test -run ^TestPipeCmds/pipe_two_commands_success$`                                                       |
| **Empty `stdout` in Pipe** | Tests a pipeline where an intermediate command succeeds but produces no output.                                                                                                                | This is a common real-world scenario (e.g., `grep "no-match" \| wc -l`). The second command should receive an empty `stdin` and execute normally, without causing an error.                     | Ensures that the piping logic is robust and can handle empty data streams, which prevents unexpected failures in otherwise valid command pipelines.                          | `TestPipeCmds/empty_stdout_in_pipe`                                                                       | Unit        | None         | `go test -run ^TestPipeCmds/empty_stdout_in_pipe$`                                                            |
| **`stderr` with Success**  | Tests a pipeline where a command writes to `stderr` (e.g., for warnings) but still exits with a success code (0).                                                                              | This test clarifies that `PipeCmds` does not capture `stderr` from intermediate commands. The pipeline should succeed, as only the exit code and final `stdout` are considered.              | It makes the behavior of the function explicit: `stderr` is not captured, which is important for developers to know so they don't rely on it for error handling.          | `TestPipeCmds/stderr_with_success`                                                                        | Unit        | None         | `go test -run ^TestPipeCmds/stderr_with_success$`                                                             |
| **Error Handling**       | Includes a comprehensive set of tests for various failure points: an error creating the pipe (`StdoutPipe`), an error starting a command (`Start`), a command not being found in the PATH, or an error waiting for a command to finish (`Wait`). | A reliable utility must handle failure at every stage. These tests ensure that errors are not swallowed but are properly propagated back to the caller.                                           | This gives developers confidence that if a command in the pipeline fails, the program will report it correctly, which is essential for debugging and creating robust applications. | `TestPipeCmds/error_in_stdoutpipe`<br>`TestPipeCmds/error_in_start`<br>`TestPipeCmds/error_in_wait`<br>`TestPipeCmds/grep_not_found` | Unit        | None         | `go test -run ^TestPipeCmds/error_in`                                                                         |
